# Grant Strategy Memory Knowledge

## Purpose

This knowledge boundary tells Codex stages how to use MAG strategy memory without turning memory transport into grant truth.

Strategy memory is Markdown-first advisory context. It gives Codex prior fundability and reviewer-risk experience to reason with; it is not a programmatic grant recipe engine, fundability scorer, route controller, quality/export gate, or submission-readiness gate.

## Owned Content

- Strategy memory bodies are MAG-owned and workspace/runtime-state backed.
- Memory may describe funder fit, prior reviewer risks, applicant evidence, successful framing, rejected directions, and package lessons.
- Repo source carries descriptor refs, locator templates, consumed-memory proof refs, writeback proposal refs, and receipt refs only.

## Stage Use

- Intake may use memory to find known call/applicant context, but must verify against current source refs.
- Fundability may accept, reject, or mark memory stale based on current call fit and reviewer risk.
- Aims and authoring may use accepted memory as strategy guidance, never as substitute evidence.
- Review may use memory to detect repeated weaknesses or prior closure patterns.
- Missing or conflicting strategy memory never blocks ordinary stage work. Record it as stale/rejected context; only a wrong-target identity/currentness, authority/safety, irreversible action, or explicit owner gate can hard-stop.

## Accept/Reject Standard

- Accept memory only when it is relevant to the locked call, applicant context, and current proposal route.
- Reject memory when it conflicts with current source refs, encourages an unsafe funder switch, overstates evidence, or hides reviewer risk.
- Mark memory stale when its factual basis may have drifted and no current source ref confirms it.

## Forbidden Uses

- Do not store memory bodies in `agent/`, `contracts/`, OPL generated surfaces, or docs as machine truth.
- Do not let memory override funding-call guidance, eligibility, source evidence, or owner receipts.
- Do not claim memory acceptance from locator success alone.
- Do not turn accepted memory into a numeric fundability score, default route, package/export verdict, or submission-ready claim.

## Receipt Expectations

- Any accepted/rejected memory must produce a body-free ref, decision state, source refs, reason, and MAG owner receipt or typed blocker.
- Memory writeback proposals must preserve funding-call lock, applicant evidence context, reviewer risk, and quality closure semantics.

## Review Questions

- Does the memory still match the current call and applicant context?
- Does it improve fundability or quality reasoning, or does it merely sound familiar?
- Would using this memory hide a current source gap or reviewer risk?
